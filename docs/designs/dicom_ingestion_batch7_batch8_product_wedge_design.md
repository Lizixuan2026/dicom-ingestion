## Problem Statement

The current DICOM ingestion system has a real backend engine: raw byte preservation, scanning, parsing, canonical persistence, duplicate detection, query/projection/replay primitives, observability, and tests.

The unresolved product problem is not “how do we parse DICOM.” The real problem is:

> How do we turn this ingestion engine into the medical-imaging intake layer of a data management platform without letting PACS, external hospital integration, or generic DICOMweb scope creep into the product?

The system must let platform users bring messy imaging data into a governed data platform, preserve source truth, extract enough semantic structure to organize and reuse it, support cloud and local/NAS deployments, and expose the results to dataset, annotation, and review workflows.

## Existing Direction From Prior Review

The prior synthesis at `reviews/codex_review/dicom_ingestion_batch7_batch8_scope_synthesis_v0.1.md` established the key sequencing:

```text
Batch 7 = Data Platform Foundation Closure
Batch 8 = Product Surface + Review Workflow Closure
Batch 9 = OHIF Bridge, only when needed
```

This office-hours pass accepts that direction and sharpens it into a product wedge and implementation posture.

## Hard Scope Boundary

This project is not PACS.

Explicitly not in scope:

- PACS compatibility
- external hospital system integration
- generic DICOMweb conformance
- implementing STOW-RS / QIDO-RS / WADO-RS as product goals
- accepting networking or hospital-protocol scope because “the standard says so”

Only allowed exception:

- a future minimal OHIF bridge, if a concrete viewer/annotation workflow requires it.

The wording matters. This is not “暂时不优先.” It is “not the product.”

## Premises

1. **The product is a data-management intake layer, not a DICOM infrastructure clone.**  
   If this premise changes, the architecture changes completely. But based on current direction, this premise should stay fixed.

2. **Batch 7 must come before Batch 8 because public APIs should not freeze false internal contracts.**  
   Parser semantics, local/NAS storage paths, folder ingest, async parsing, and binding vocabulary are still too implicit. Exposing APIs first would turn those implicit choices into migration debt.

3. **Private tag semantics are not optional because storage layout depends on them.**  
   `MeasUID` and related vendor/device metadata are not “nice-to-have metadata.” They drive local/NAS organization and later human browsability.

4. **Folder/tree ingest is part of the platform promise, not upload polish.**  
   A platform intake layer that only supports single files and ZIPs does not match the way imaging data is commonly handed around in local research/data environments.

5. **Series-level review is the right user abstraction for duplicate/conflict UX.**  
   SOP-level evidence is necessary internally, but users should not resolve batch duplicates by reading hundreds of instance-level findings.

6. **Celery should be an adapter, not the domain boundary.**  
   The product needs async parsing and observable job state. Celery may be the first implementation, but the domain should expose a parse-task boundary that could survive infrastructure changes.

## Approaches Considered

### Approach A: Product Surface First

Summary: Implement upload/query/conflict APIs immediately on top of the current service layer, then backfill parser, storage, and async improvements later.

Effort: M  
Risk: High

Pros:

- Gives users and downstream services visible endpoints sooner.
- Reuses current services with less immediate refactor pressure.
- Makes demos easier because there is an API surface to show.

Cons:

- Freezes weak internal contracts before parser/storage semantics are mature.
- Makes later local/NAS path generation and async parsing changes harder.
- Encourages treating storage, private tags, and folder ingest as afterthoughts.

Reuses:

- Current query/projection/replay/conflict services.

Verdict: Reject as default. This optimizes for demo surface, not durable platform correctness.

### Approach B: Foundation First, Then Product Surface

Summary: Batch 7 closes parser semantics, tag schema, dual storage, folder/tree ingest, async parse boundary, and binding vocabulary. Batch 8 exposes APIs, review workflow, conflict resolution, and operator handoff.

Effort: L  
Risk: Medium

Pros:

- Makes the platform contracts true before exposing them.
- Directly addresses the human review findings Q2-Q7.
- Keeps future Batch 8 APIs cleaner and less likely to need migration.
- Preserves a clear product identity: data platform intake layer, not PACS.

Cons:

- Slower path to a visible API demo.
- Requires stronger architecture discipline in Batch 7.
- Needs explicit decisions for path fallback, parser interface depth, and queue abstraction.

Reuses:

- Upload persistence, scanner, canonical persistence, duplicate detection, conflict service, projections, observability, and runbook primitives.

Verdict: Recommended. This is the best match for the stated product direction.

### Approach C: Local/NAS Storage Wedge First

Summary: Treat local/NAS organization as the sharpest wedge. Prioritize tag-derived path generation, private tags, folder ingest, and human-browsable output before broader parser/async/API closure.

Effort: M  
Risk: Medium

Pros:

- Produces a concrete user-visible improvement quickly: files become organized in a predictable medical-imaging hierarchy.
- Forces private-tag semantics to become real early.
- Avoids drifting into generic ingestion abstractions.

Cons:

- Could overfit to one deployment/storage mode if not balanced with object storage.
- Does not by itself close product APIs or conflict resolution.
- Needs careful boundary so path layout does not become the whole domain model.

Reuses:

- Current raw object persistence and parser output, but requires parser/tag upgrades.

Verdict: Strong sub-wedge inside Approach B, not the whole plan.

## Recommended Approach

Choose **Approach B: Foundation First, Then Product Surface**.

The strategic framing is now locked:

> Batch 7 is the **Local/NAS Folder Ingest Wedge**.
> Foundation Closure is the implementation scope that makes that wedge true.

This distinction matters. “Foundation Closure” is an engineering task list. “Local/NAS Folder Ingest Wedge” is the product proof. It keeps the work pointed at the user-visible outcome: a real folder/tree of medical imaging data goes in, governed platform storage comes out, and the system can explain what happened.

The minimum Batch 7 proof is:

```text
Given a realistic folder/tree input,
when the ingest job runs,
then the platform preserves original DICOM bytes,
extracts configured standard and private tags,
places data through the selected object-storage or local/NAS backend,
tracks async parse state,
and produces an ingest report that explains accepted files, rejected files, duplicates, missing tags, fallback paths, and storage destinations.
```

If Batch 7 proves that, Batch 8 APIs become straightforward wrappers around real platform behavior rather than optimistic promises.

## Batch 7 Product Wedge

Batch 7 should be narrated as **Local/NAS Folder Ingest Wedge** and implemented as **Data Platform Foundation Closure**.

Required workstreams:

1. Parser seam + tag schema
2. Private tag semantics, especially storage-relevant tags like `MeasUID`
3. Dual storage backend abstraction
4. Deterministic local/NAS path generator
5. Folder/tree ingest model
6. Async parse worker boundary
7. Removal of ambiguous `BindingTargetType.STUDY`

The output of Batch 7 should be demonstrable without a rich UI:

```text
Given a realistic folder/tree input,
when the ingest job runs,
then the platform preserves original bytes,
extracts configured tags,
places data using the selected storage backend,
tracks async parse state,
and produces a report that explains what happened.
```

That is the minimum proof that this is a platform intake layer.

## Batch 8 Product Surface

Batch 8 should be framed as **Product Surface + Review Workflow Closure**.

Required workstreams:

1. Upload/job APIs
2. Study/Series/Instance query APIs
3. Series-level duplicate summaries
4. Series conflict resolution workflow
5. Platform binding response envelope
6. Targeted operator runbooks for stuck batch, replay, and projection rebuild

The output of Batch 8 should prove:

```text
A user or downstream service can drive ingestion, inspect results,
understand duplicate/conflict state at Series level,
resolve conflicts safely,
and recover operationally without reading service internals.
```

## Open Decisions

These decisions should be resolved before implementation begins.

### 1. Local/NAS fallback path rules

Decision needed: what path do we generate when vendor, device name, `MeasUID`, or other expected tags are missing?

Recommended default:

```text
DICOM_{MODALITY}/UNKNOWN_VENDOR/UNKNOWN_DEVICE/{StudyUID}/NO_MEAS_UID/{SeriesUID}/{SOPInstanceUID}.dcm
```

But this needs domain confirmation. The wrong fallback becomes filesystem clutter forever.

### 2. Parser interface depth

Decision needed: should the first parser seam include only `parse_header`, or also tag processors/private dictionaries?

Recommended default: include tag schema and processors now. Storage design already depends on semantic private tags, so a shallow `parse_header` wrapper would be fake abstraction.

### 3. Async queue boundary

Decision needed: direct Celery design or domain queue interface with Celery adapter?

Recommended default: domain queue interface with Celery adapter. This keeps Celery out of the core model while still shipping async behavior.

### 4. Folder ingest representation

Decision needed: API folder upload as real server-side directory, uploaded ZIP with preserved relative paths, or manifest-based tree?

Recommended default:

- local/internal CLI: real directory path allowed,
- external/API boundary: ZIP or manifest with preserved relative paths,
- scanner sees a common `InputTree` abstraction.

This avoids pretending browser/API clients can literally upload arbitrary local folders without a transport shape.

### 5. Multimodal scope in Batch 7

Decision needed: implement all modality path structures now, or reserve grammar and implement DICOM-first?

Recommended default: reserve grammar, implement DICOM-first. Full multimodal storage is likely an ocean. DICOM storage foundation is a lake.

## Success Criteria

Batch 7 is successful when:

- parser callers depend on an interface, not pydicom directly,
- tag schema supports standard and private semantic tags,
- storage backend abstraction supports object storage and local/NAS modes,
- local/NAS path generation is deterministic and tested,
- folder/tree ingest works through a common input abstraction,
- parse work has real `IN_PROGRESS`, retry, failed, and completed states,
- `BindingTargetType.STUDY` is removed or renamed before further product surface expands.

Batch 8 is successful when:

- users/downstream services can call stable upload/job/query APIs,
- duplicate evidence is summarized at Series level,
- conflicts can be resolved with idempotent repeat behavior and 409 on conflicting actions,
- platform binding result is visible without hiding valid DICOM on binding failure,
- targeted runbooks exist for stuck batch, replay, and projection rebuild.
